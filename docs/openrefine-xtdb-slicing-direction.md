# OpenRefine × XTDB スライシング方向性メモ

OpenRefine にソースコードを入れて探索しつつ、XTDB に蓄積した構造化分析結果を活用して
変更影響分析やスライシングを進めるための方向性メモ。

README や API ドキュメントとは別に、設計判断の軸を見失わないための記録として残す。

![OpenRefine と XTDB と Clojure の役割分担、およびスライシングの処理ループ](assets/openrefine-xtdb-slicing-loop.svg)

---

## 1. 背景

これまで OpenRefine 上では、インテントと文字列パターンを手掛かりに
ソースコードを探索する運用が中心だった。

この方法でも一定の成果は出るが、以下の限界がある。

- 慣れても探索に時間がかかる
- 呼び出し関係や型情報を人手で補う場面が多い
- スライシングのような依存追跡で、構造情報を十分に活かしきれない

一方で `openrefine-workbench` にはすでに、XTDB 上に以下のような分析結果を蓄積する仕組みがある。

- AST 相当の取り込み結果
- `xref` / `jref` による参照関係
- `jsig` によるシグネチャ情報
- `call-tree` や依存グラフ
- `jacoco` や `sqlref` などの補助的な解析結果

つまり、これからの主戦場は「文字列パターンを工夫すること」ではなく、
「XTDB にある意味情報を OpenRefine に持ち込み、人間の判断と組み合わせること」にある。

---

## 2. 基本方針

### OpenRefine は探索層

OpenRefine は生データや中間結果を GUI で観察し、
絞り込み、分類し、人間の判断を書き込むための場所と捉える。

強みは次のような操作にある。

- facet による分類
- filter による絞り込み
- clustering / grouping による意味づけ
- cross による複数表の突き合わせ
- 除外・タグ付け・優先度付けのような人間判断の反映

### XTDB は永続層

XTDB はソースコードから抽出した構造化情報を保持し、
再利用可能な形でクエリ可能にする永続層と捉える。

OpenRefine のために毎回ゼロから解析するのではなく、
一度取り込んだ情報を何度でも別の切り口で使い回す。

### Clojure は計算層

再帰的な依存探索や到達可能性計算のような重い処理は、
OpenRefine 側ではなく Clojure / XTDB 側で行う。

OpenRefine には「計算済みの見やすい結果」を渡す。

---

## 3. 目指すアーキテクチャ

三者の役割分担は次の通り。

```text
Clojure        = スライス計算エンジン
XTDB           = スライス永続層 / 意味情報ストア
OpenRefine     = スライス探索 UI / 編集 UI
```

処理の流れは次のループを基本とする。

```text
1. ソースコードを ingest / xref / jref / jsig で XTDB に投入
2. Clojure で call graph / xref / type 情報からスライス候補を計算
3. 結果を CSV / JSON / EDN として OpenRefine に渡す
4. OpenRefine で人間が探索・除外・タグ付け・優先度付けを行う
5. 必要ならその判断結果を trial に戻して再計算する
```

このループにより、自動計算と人間判断を分離しつつ接続できる。

なお、ここでいう中間結果は OpenRefine 専用形式である必要はない。
TSV / CSV / JSON のような平坦表として出しておけば、
OpenRefine だけでなく Excel や AI Agent のチャット文脈でも使い回せる。

特に call-flow のようなスライス結果は、
ソース全文をそのまま渡す前の「圧縮済みコンテキスト」として有効であり、
人間にも AI にも読みやすい。

## 4. スライシングの考え方

この文書でいうスライシングは、一般的な program slicing のような
「依存関係を自動展開した完成結果」だけを指さない。

むしろ、変数・メソッド・責務・副作用などの流れを分離し、
人間が追跡・除外・タグ付けしながら意味づけしていく探索ループを指す。

これは、自動展開が不要という意味ではない。
むしろ `xref` や call graph のような自動計算は重要な出発点である。

ただし実際の調査では、自動計算だけではやりきれないことが多い。
理由は次の通り。

- 依存として到達可能でも、今回の調査目的には無関係なものが混ざる
- 逆に、責務境界や副作用のように構造情報だけでは拾いきれない流れがある
- 調査の途中で「何が足りないか」を事前に予測しきれない
- 最終的には、人間が調査対象とノイズを切り分ける必要がある

したがって、スライシングは本質的には「依存関係のフィルタリングと展開」であると同時に、
「自動計算の結果を人間が再編集していく探索プロセス」でもある。

例えば、あるメソッドを起点にして以下を求めたい。

- 直接呼び出している先
- 間接的に影響する先
- 逆に誰から参照されているか
- どのファイル / クラス / メソッドに属するか
- SQL / テスト / カバレッジなど他の分析結果とどう重なるか

これらは文字列走査よりも、`xref` や call graph を使う方が自然で精度も高い。
ただし、その結果をそのまま完成品として受け取るのではなく、
「どの流れを追いたいのか」という視点で再整理できることが重要になる。

したがって、OpenRefine でスライシングしたい場合も、
OpenRefine 上でグラフ探索を頑張るのではなく、
XTDB 側で展開した結果をテーブルとして持ち込む方針を取る。

OpenRefine 側で重視するのは、完成済みのスライスを眺めることではなく、
必要な流れを分離しながら追跡の視点を切り替えられることである。

---

## 5. OpenRefine に持ち込む粒度

今後の重要な設計判断は、「何を基本単位として OpenRefine に渡すか」である。

候補は主に次の 3 つ。

- `file` 単位
- `line` 単位
- `method` 単位

当面は `method` 単位を主軸にするのが有力。

理由は次の通り。

- スライシングや影響範囲分析はメソッド粒度と相性が良い
- `call graph` や `xref` の結果をそのまま乗せやすい
- `line` 単位より表が安定しやすく、UI 上でも扱いやすい

必要なときだけ `line` 単位に降りる。
つまり、普段は `method` ベースで探索し、詳細確認でソース行を参照する。

---

## 6. テーブル設計の基本方針

OpenRefine には 1 枚の巨大テーブルを渡すより、役割別に分けた複数表を持ち込む方がよい。

例:

- `source-lines`
  - `file`, `line`, `text`
- `methods`
  - `method-id`, `class`, `file`, `start-line`, `end-line`, `signature`
- `calls`
  - `from-method`, `to-method`, `depth`, `kind`
- `xref`
  - `from`, `to`, `file`, `line`
- `slice-results`
  - `root-method`, `method-id`, `depth`, `direction`, `reason`

この分割により、OpenRefine 側では `cross()` を使って必要な情報を重ねられる。

実際には、最初から完全に分割した複数表で始めなくてもよい。
当面の call-flow では、次のような平坦表 1 枚でも十分に価値がある。

- `slice-call-flow`
  - `root-method`
  - `method-id`
  - `class`
  - `method`
  - `depth`
  - `parent-method`
  - `method-file`
  - `method-start-line`
  - `method-end-line`
  - `call-file`
  - `call-line`

この形なら、

- OpenRefine に読み込んで facet / filter する
- Excel で深さや親子関係を見る
- AI Agent に渡す前の圧縮コンテキストにする

といった使い方を同じデータで兼用できる。

### join key の重要性

ソースと分析結果を一緒に扱うには、結合キーを先に固定する必要がある。

候補:

- `method-id`
- `file + line`
- `file + start-line + end-line`

当面は `method-id` を主キーにしつつ、
ソース表示用には `file + line` 系のキーを補助的に使う形が扱いやすい。

---

## 7. ソースと分析結果を一緒に扱う

これは十分可能であり、このワークベンチの二層構造とも整合する。

例えば次のような形が考えられる。

### ソースコード表

| file | line | text |
|---|---:|---|
| `Foo.java` | 10 | `public void run() {` |
| `Foo.java` | 11 | `Util.x();` |

### 分析結果表

| file | line | method-id | ast-type | calls | called-by | depth |
|---|---:|---|---|---|---|---:|
| `Foo.java` | 11 | `Foo#run` | `MethodCall` | `Util#x` |  | 1 |

この 2 つを `file` や `line`、あるいは `method-id` で結ぶことで、
「この行はどのメソッドに属するか」
「この行はどこを呼んでいるか」
「この行は変更影響スライスに含まれるか」
を OpenRefine 上で確認できる。

このとき source 側の対象範囲は、必ずしも `root + depth` だけで決めなくてよい。
実務ではむしろ、

- `:scope :all` で `:input/java-roots` 全体を出す
- `:package-prefixes` で特定 package 配下に絞る
- `:classes` で特定 class 群だけを出す
- `:files` で明示的な source file 一覧を出す

といった source 主導の scope の方が扱いやすいことが多い。
`root + depth` は、その scope 上に call-flow の印を重ねる補助情報として使うのが自然である。

---

## 8. OpenRefine に期待する役割

OpenRefine に期待するのは、依存グラフを計算することではない。
計算結果を人間が読める形で探索し、編集し、判断を加えることである。

これは、デバッガ上で変数や責務の流れを追いたいときの
「この流れだけを切り出して見たい」という操作感に近い。
OpenRefine は、その操作感を構造化データの上で再現する探索 UI と捉えられる。

特に有効なのは次のような用途。

- depth ごとの facet でスライスの広がりを見る
- class / file / module 単位で偏りを確認する
- 外部依存やノイズを除外する
- 重要ノードにタグを付ける
- 「この範囲は今回の調査対象外」といった判断を残す
- 不足している観点や追加で追うべき流れを発見する

つまり OpenRefine は、コード閲覧ツールというより
「分析結果に対する人間の編集 UI」として位置付ける。

## 9. 当面の実装指針

最初から全部つなぐより、次の順で小さく始める。

最初の段階では、すべての flow を同時に扱おうとしない。
まずは「何の流れを slice したいのか」を 1 つか 2 つに絞って始める。

候補としては次のような flow がある。

- 呼び出しの流れ
- 値の流れ
- 副作用の流れ
- SQL / I/O の流れ
- 責務の流れ

この選択によって、XTDB 側で集めるべき関係と、
OpenRefine に持ち込む列構成は変わる。

当面は、既存の `call graph` や `xref` を活かしやすい
「呼び出しの流れ」を最初の主対象にするのが現実的である。
その上で、必要に応じて値の流れや副作用の流れを追加していく。

### Step 1. method 単位の最小スキーマを決める

最低限ほしい列:

- `method-id`
- `class`
- `file`
- `start-line`
- `end-line`
- `depth`
- `calls`
- `called-by`

### Step 2. XTDB からスライス結果を出すクエリを作る

まずは「起点メソッドから深さ N までの呼び出し先一覧」を
CSV / JSON 化できれば十分な価値がある。

ここで重要なのは、完全な答えを 1 回で出すことではなく、
OpenRefine 上で追跡を始められる素材を安定して出せるようにすることである。

当面の実装としては、`run-trial` から `:analyze/slice-call-flow` を実行し、
`exports/*.tsv` に平坦表として書き出せる形が扱いやすい。
これにより、OpenRefine に持ち込む前の段階でも、
Excel や AI Agent で slice を確認できる。

### Step 3. OpenRefine で読みやすい表にする

1 回目から完璧な正規化を狙わず、
facet / filter で使いやすい列構成を優先する。

特に、除外・優先度・対象外・追跡メモのような
人間判断を書き込める余地を最初から持たせておくと、
後で `trial` に戻す設計につなげやすい。

### Step 4. 人間の判断を戻す経路を考える

将来的には OpenRefine 側で付けたタグや除外指定を
`trial.edn` や別の中間ファイルに戻せるようにする。

---

## 10. やらないこと / 気をつけること

### OpenRefine に計算責務を載せすぎない

再帰的な到達可能性計算や call graph 展開は Clojure / XTDB 側で行う。
OpenRefine はその結果を読む場所に留める。

### 1 枚の巨大表に寄せすぎない

何でも 1 テーブル化すると運用が苦しくなる。
粒度と責務に応じて表を分ける。

### 文字列探索を完全には捨てない

文字列パターン探索は初期の当たり付けや例外処理でまだ有効。
ただし主軸は「意味情報ベース」に移す。

---

## 11. この方向性の要約

今後の中心は次の一文に集約できる。

> OpenRefine にソースを入れること自体が主役ではない。  
> XTDB で計算した意味情報に、OpenRefine で人間の判断を重ねることが主役である。

この方針に沿えば、
自動計算は Clojure / XTDB に寄せ、
探索と判断は OpenRefine に寄せるという役割分担が明確になる。

結果として、スライシングや変更影響分析は
「文字列パターン中心の探索」から
「構造化分析結果を使った再利用可能な探索」へ進化できる。
